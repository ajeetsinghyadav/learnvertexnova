# 05: Camera Interaction and vneinteraction

**vneinteraction** sits on top of vnescene ŌĆö it takes an `ICamera` from vnescene and drives it from mouse, keyboard, and touch events. It does not own geometry or manage a window ŌĆö its job is solely to keep a camera's position and orientation in sync with user intent, one `update()` call per frame.

The library is at [vneinteraction](https://github.com/vertexnova/vneinteraction). The [vnetestbed](https://github.com/vertexnova/vnetestbed) `03_test_interaction` sample runs all six manipulator types interactively.

## Camera degrees of freedom

Before diving into the API, it helps to see the full motion space a camera can occupy. A camera in 3D has up to **7 degrees of freedom (DOF)**:

- **3 translation DOF** ŌĆö move along X (pan left/right), Y (pan up/down), Z (dolly in/out)
- **3 rotation DOF** ŌĆö yaw (around world-up Y), pitch (elevation), roll (around view axis)
- **1 zoom DOF** ŌĆö independent of translation: dolly moves the camera, zoom changes FOV or a virtual scale

Most manipulators expose a subset: `OrbitManipulator` and `ArcballManipulator` give you rotation + zoom (no direct position input from the user). `FpsManipulator` locks roll (5 DOF). `FlyManipulator` allows all 6 movement DOF plus zoom (7 DOF total). `OrthoPanZoomManipulator` gives 2D pan + zoom only.

![Camera degrees of freedom ŌĆö translation, rotation, zoom](/img/learn/diagrams/05_camera_dof.png)

## What vneinteraction gives you

| Component | What it is |
|-----------|-----------|
| `ICameraController` | Interface for objects that own and drive an `ICamera`: `setCamera`, `update`, `reset`, `setEnabled`. |
| `CameraSystemController` | Concrete controller: owns camera + manipulator, routes all raw input, drives `update()`. |
| `ICameraManipulator` | Interface for all manipulators ŌĆö `handle*` input methods, `applyCommand`, `fitToAABB`, `resetState`. |
| `CameraManipulatorFactory` | Creates any manipulator by `CameraManipulatorType`. One `create(type)` call. |
| `OrbitManipulator` | Spherical orbit around a center of interest: yaw/pitch rotation, zoom, pan, inertia. |
| `ArcballManipulator` | Virtual trackball: quaternion rotation, zoom, pan, inertia. |
| `FpsManipulator` | First-person: WASD + Q/E + right-mouse look. Horizon stays level (world-up locked). |
| `FlyManipulator` | 6-DOF free flight: WASD + Q/E with no world-up constraint. Camera can pitch past vertical. |
| `OrthoPanZoomManipulator` | 2D pan and zoom for orthographic views. No rotation. `supportsPerspective()` returns `false`. |
| `FollowManipulator` | Autonomous follow / tracking ŌĆö camera chases a target with exponential damping. Implements `ICameraBehavior`. |
| `CameraInputBindings` | Configurable mouse-button and key mappings used by the input adapter inside `CameraSystemController`. |
| `CameraActionType` | Semantic command layer: intent-based commands between input source and manipulator ŌĆö remappable and testable without a window. |

Full reference: **[vneinteraction repository](https://github.com/vertexnova/vneinteraction)**.

## Architecture ŌĆö controller, manipulator, camera

Three responsibilities are kept separate so each piece can be tested and swapped independently.

<!-- ![vneinteraction three-layer architecture](/img/learn/diagrams/05_interaction_architecture.png) -->

**1. `CameraSystemController`** owns the active camera and the active manipulator. Each frame you call `controller.update(delta_time)`. When input arrives you call `controller.handleMouseMove(...)`, `handleMouseButton(...)`, `handleMouseScroll(...)`, `handleKeyboard(...)`, `handleTouchPan(...)`, or `handleTouchPinch(...)`. The controller forwards each call to the manipulator.

**2. `CameraInputAdapter`** (internal) sits inside `CameraSystemController`. It translates raw input ŌĆö button indices, pixel coordinates, key codes ŌĆö into `CameraActionType` semantic commands using the active `CameraInputBindings`. This decouples the manipulator from the physical button layout. You can remap bindings without touching any manipulator code.

**3. The manipulator** consumes `CameraActionType` payloads and updates the camera. It holds a `std::shared_ptr<ICamera>` and writes position, target, and up through that handle. It does not own the camera ŌĆö you can hand the same camera to a different manipulator at any time by calling `controller.setManipulator(...)`.

## Manipulator hierarchy

<!-- ![Manipulator class hierarchy](/img/learn/diagrams/05_manipulator_hierarchy.png) -->

`OrbitStyleBase` shares the orbit/pan/zoom/inertia machinery used by both `OrbitManipulator` and `ArcballManipulator` ŌĆö that is why `setRotationDamping()` and `setPanDamping()` appear on both. `FreeCameraBase` shares the WASD/look logic between `FpsManipulator` and `FlyManipulator`. The only behavioral difference between them is what `upVector()` returns (see [┬¦ FPS and Fly](#free-camera-movement--fps-and-fly)).

`OrthoPanZoomManipulator::supportsPerspective()` returns `false` ŌĆö assigning a `PerspectiveCamera` to it is a no-op. `FollowManipulator` additionally implements `ICameraBehavior`, making it usable without an input pipeline entirely.

## Orbit and arcball ŌĆö center of interest model

Both `OrbitManipulator` and `ArcballManipulator` rotate around a **center of interest (COI)** ŌĆö a world-space pivot point. The camera is always positioned at a fixed `distance` from the COI. The two manipulators differ only in how they represent the rotation.

### Spherical coordinates and COI

![Orbit ŌĆö spherical coordinates around COI](/img/learn/diagrams/05_orbit_spherical.png)

`OrbitManipulator` stores `yaw_deg` (azimuth, rotation around world-up Y) and `pitch_deg` (elevation above the equatorial plane). The camera position is derived from spherical coordinates:

$$
\mathbf{d} =
\begin{bmatrix}
\cos\theta \sin\phi \\
\sin\theta \\
\cos\theta \cos\phi
\end{bmatrix}
\qquad
\text{position} = \text{COI} + r\,\mathbf{d}
$$

where $\phi$ = `yaw_deg` in radians, $\theta$ = `pitch_deg` in radians, and $r$ = `distance`. The view matrix is then built as a standard lookAt from `position` toward `COI` with `world_up` ŌĆö exactly the construction covered in [chapter 04 ┬¦&nbsp;View matrix construction](./04-scene-and-vnescene#view-matrix-construction).

The internal state:

```cpp
// OrbitCameraState (interaction_types.h)
struct OrbitCameraState {
    Vec3f coi_world;     // pivot point in world space
    float distance;      // radius of the orbit sphere, default 5.0
    Vec3f world_up;      // default (0, 1, 0)
    float yaw_deg;       // azimuth, 0┬░ = front
    float pitch_deg;     // elevation, clamped to avoid gimbal flip
};
```

`pitch_deg` is clamped near ┬▒90┬░ so the camera never flips past the poles.

### Arcball ŌĆö quaternion virtual trackball

![Arcball ŌĆö virtual trackball projection](/img/learn/diagrams/05_arcball_projection.png)

`ArcballManipulator` projects mouse pixel positions onto a virtual unit sphere overlaid on the viewport, then derives a quaternion rotation from the arc swept between drag-start and drag-end.

Mapping a pixel $(x, y)$ in a viewport of size $(W, H)$ to normalized device coordinates:

$$
n_x = \frac{2x}{W} - 1
\qquad
n_y = 1 - \frac{2y}{H}
$$

Project onto the sphere surface (inside the circle) or the hyperbolic sheet (outside the unit disk):

$$
r^2 = n_x^2 + n_y^2
\qquad
\mathbf{p} =
\begin{cases}
(n_x,\; n_y,\; \sqrt{1 - r^2}) & r \leq 1 \\
\text{normalize}(n_x,\; n_y,\; 0) & r > 1
\end{cases}
$$

The rotation quaternion from drag-start $\mathbf{p}_0$ to drag-end $\mathbf{p}_1$:

$$
\mathbf{axis} = \mathbf{p}_0 \times \mathbf{p}_1
\qquad
\cos(\alpha/2) = \mathbf{p}_0 \cdot \mathbf{p}_1
\qquad
q = \bigl(\sin(\alpha/2)\,\hat{\mathbf{axis}},\; \cos(\alpha/2)\bigr)
$$

This quaternion is accumulated into the arcball rotation state: $q_{\text{new}} = q \cdot q_{\text{prev}}$. The camera position is then $\text{COI} + q_{\text{new}} \cdot \mathbf{d}_0 \cdot r$.

**Orbit vs arcball:** Orbit tracks explicit Euler yaw/pitch ŌĆö pitch can be clamped and `setViewDirection()` can snap to named presets. Arcball accumulates a quaternion ŌĆö any orientation is reachable with no gimbal risk, but there is no natural "up" enforcement and no view-direction snapping.

The `ArcballCameraState` stores `coi_world`, `distance`, and the accumulated `rotation` as a `Quatf`.

## Zoom methods

![Zoom methods comparison](/img/learn/diagrams/05_zoom_methods.png)

All orbit-family and free-camera manipulators support three zoom strategies via `setZoomMethod(ZoomMethod)`:

| `ZoomMethod` | What changes | Use when |
|---|---|---|
| `eDollyToCoi` | Camera translates toward/away from COI along the view ray | Natural 3D exploration ŌĆö default for most scenes |
| `eSceneScale` | `scene_scale_` multiplier accumulates; COI and camera unchanged | Map viewers, top-down editors, orthographic layouts |
| `eChangeFov` | Perspective FOV angle widens or narrows | Cinematic zoom-lens effect; depth compression/dilation |

`setFovZoomSpeed(float)` controls the step size when `eChangeFov` is active. `getSceneScale()` returns the accumulated `eSceneScale` factor (starts at 1.0).

**Pan world-units-per-pixel** ŌĆö used internally so that a one-pixel pan drag always moves the scene by the same apparent amount regardless of zoom level:

$$
\text{worldUnitsPerPixel} = \frac{\text{sceneScale} \times \text{distance}}{\text{viewportHeight}}
$$

Call `getWorldUnitsPerPixel()` on any manipulator when you need to convert pixel deltas to world-space units yourself.

## Free-camera movement ŌĆö FPS and Fly

![Free camera ŌĆö yaw/pitch look model](/img/learn/diagrams/05_free_camera_look.png)

Both `FpsManipulator` and `FlyManipulator` represent camera orientation as a yaw angle $\phi$ and a pitch angle $\theta$. The forward vector is:

$$
\mathbf{f} =
\begin{bmatrix}
\cos\theta_{\text{rad}}\,\cos\phi_{\text{rad}} \\
\sin\theta_{\text{rad}} \\
\cos\theta_{\text{rad}}\,\sin\phi_{\text{rad}}
\end{bmatrix}
$$

Each frame, WASD + Q/E key states produce a signed movement direction. The world-space position delta is:

$$
\Delta\mathbf{p} = \bigl(\mathbf{f} \cdot \text{fwd} + \mathbf{r} \cdot \text{right} + \mathbf{u} \cdot \text{up}\bigr) \times \text{effectiveSpeed} \times \Delta t
$$

where fwd, right, up $\in \{-1, 0, +1\}$ from keys held, and:

$$
\text{effectiveSpeed} = \text{moveSpeed} \times m, \qquad
m = \begin{cases} \text{sprintMultiplier} & \text{Shift held} \\ \text{slowMultiplier} & \text{Ctrl held} \\ 1.0 & \text{otherwise} \end{cases}
$$

**FPS vs Fly:** `FpsManipulator` always derives the right vector as $\mathbf{r} = \text{normalize}(\mathbf{f} \times \mathbf{worldUp})$ ŌĆö the horizon stays level, Q/E move strictly up/down in world space, and `pitch_deg` is clamped to ┬▒89┬░. `FlyManipulator` derives the right vector from the camera's current local up (no world-up constraint), so Q/E move in camera-local up, pitch is unclamped, and the camera can roll past vertical ŌĆö useful for space or flight simulators.

The internal state is `FreeCameraState` (`position`, `yaw_deg`, `pitch_deg`, `up_hint`).

## OrthoPanZoom ŌĆö 2D interaction

`OrthoPanZoomManipulator` is a pan-and-zoom-only manipulator for `OrthographicCamera`. It has no rotation at all.

`supportsPerspective()` returns `false`. Calling `setCamera()` with a `PerspectiveCamera` is silently ignored. Always pair it with `vne::scene::OrthographicCamera`.

**Pan** translates the orthographic bounds (`left/right/bottom/top`) by a world-space delta computed from `getWorldUnitsPerPixel()`. The cursor world position is preserved after zoom ŌĆö a standard zoom-to-cursor operation. If the cursor is at world position $\mathbf{c}$ and the zoom factor is $z$:

$$
\text{newOffset} = \text{currentOffset} + (\mathbf{c} - \text{cameraCenter}) \times (1 - 1/z)
$$

This shifts the camera center so the point under the cursor appears to stay fixed.

Inertia is applied to pan through `setPanDamping(float)` (default 10.0). Higher values decay velocity faster (snappier stop). Setting damping to 0 disables inertia.

## Follow / tracking camera

`FollowManipulator` is autonomous ŌĆö no mouse or keyboard input routes to it. The `handle*` methods are all no-ops. You set a target and the camera chases it each `update()`.

Two modes:

```cpp
// Static target ŌĆö camera chases a fixed world position
follow->setTargetWorld({0.0f, 1.0f, 0.0f});

// Dynamic target ŌĆö callback evaluated every update()
follow->setTargetProvider([&entity]() {
    return entity.getPosition();
});
```

Camera position is updated using exponential smoothing (frame-rate-independent):

$$
\mathbf{pos}_{\text{new}} = \text{lerp}\!\bigl(\mathbf{pos}_{\text{current}},\; \mathbf{target} + \mathbf{offset},\; 1 - e^{-\,\text{damping}\,\cdot\,\Delta t}\bigr)
$$

Using $1 - e^{-\,k\,\Delta t}$ rather than a fixed lerp factor ensures the same perceived lag regardless of frame rate. At 30 fps or 120 fps the camera appears to close the same fraction of the remaining gap per unit of wall-clock time.

`setOffsetWorld(Vec3f)` controls where the camera sits relative to the target (default `{0, 0, 5}`). `setDamping(float)` (default 8.0) controls how quickly the camera catches up ŌĆö higher = snappier.

## Conventions

| Term | Meaning |
|------|---------|
| COI | Center of interest ŌĆö world-space pivot that orbit and arcball rotate around |
| `distance` | Radius of the orbit sphere ŌĆö camera's distance from COI, default 5.0 |
| `yaw_deg` | Azimuth rotation around world-up. 0┬░ = looking in -Z direction (front) |
| `pitch_deg` | Elevation. Positive = looking up. Clamped near ┬▒90┬░ for orbit/FPS |
| `scene_scale_` | Accumulated zoom factor when `ZoomMethod::eSceneScale` is active (starts at 1.0) |
| `delta_time` | Frame time in seconds. All movement, damping, and inertia are multiplied by this |
| `worldUnitsPerPixel` | World space covered by one screen pixel at current zoom level |

Default input bindings (`CameraInputBindings` from `interaction_types.h`):

| Input | Orbit / Arcball | FPS / Fly |
|-------|-----------------|-----------|
| LMB drag | Rotate | ŌĆö |
| RMB drag | Pan | Right-mouse look (hold to activate) |
| MMB drag | Pan | ŌĆö |
| Scroll | Zoom | Zoom (or dolly step) |
| W / S | ŌĆö | Forward / Backward |
| A / D | ŌĆö | Strafe left / right |
| Q / E | ŌĆö | Down / Up |
| Shift | Pan modifier (orbit) | Sprint |
| Ctrl | ŌĆö | Slow |

Bindings can be remapped via `controller.setInputBindings(CameraInputBindings)` without changing any manipulator code.

## Quick start

```cpp
#include <vertexnova/interaction/interaction.h>
#include <vertexnova/scene/scene.h>

// 1. Camera from vnescene (chapter 04)
auto camera = std::make_shared<vne::scene::PerspectiveCamera>(
    60.0f, 1280.0f, 720.0f, 0.1f, 1000.0f);
camera->setPosition({4.0f, 3.0f, 6.0f});
camera->setTarget({0.0f, 0.0f, 0.0f});

// 2. Create controller + orbit manipulator
vne::interaction::CameraManipulatorFactory factory;
vne::interaction::CameraSystemController controller;
controller.setManipulator(factory.create(vne::interaction::CameraManipulatorType::eOrbit));
controller.setCamera(camera);
controller.setViewportSize(1280.0f, 720.0f);

// 3. In your event callback (vneevents, GLFW, SDL, etc.)
//    controller.handleMouseMove(x, y, dx, dy, delta_time);
//    controller.handleMouseButton(button, pressed, x, y, delta_time);
//    controller.handleMouseScroll(sx, sy, mx, my, delta_time);
//    controller.handleKeyboard(key, pressed, delta_time);

// 4. Each frame
controller.update(delta_time);
camera->updateMatrices();

// 5. Upload to GPU (chapter 04 pattern)
auto cam_gpu = camera->toGpu();
// glBufferSubData(..., sizeof(cam_gpu), &cam_gpu);
```

Quick-reference ŌĆö key controller calls:

| Call | What it does |
|------|-------------|
| `controller.setManipulator(factory.create(type))` | Hot-swappable: reassigns current camera to new manipulator |
| `controller.setCamera(cam)` | Assigns camera to controller and to the active manipulator |
| `controller.setViewportSize(w, h)` | Tell manipulator the viewport; call again on window resize |
| `controller.update(dt)` | Advances inertia, damping, and follow-target tracking |
| `controller.reset()` | Resets manipulator state; camera position unchanged |
| `controller.setEnabled(false)` | Silences all input routing without detaching the manipulator |
| `manipulator->fitToAABB(min, max)` | Frames a bounding box ŌĆö sets orbit distance and COI |
| `manipulator->getWorldUnitsPerPixel()` | Scale-invariant pan multiplier |
| `manipulator->getSceneScale()` | Accumulated `eSceneScale` zoom factor |

### Runtime hot-swap

Swap the manipulator at any time ŌĆö same controller, same camera:

```cpp
// Switch from orbit to FPS mid-session
controller.setManipulator(factory.create(vne::interaction::CameraManipulatorType::eFps));
// setManipulator() calls assignCameraToManipulator() internally ŌĆö no need to re-call setCamera().
```

On window resize, call `setViewportSize()` again; the controller forwards it to the active manipulator.

## CameraSystemController ŌĆö full API reference

```cpp
#include <vertexnova/interaction/camera_system_controller.h>
```

| Method | What it does |
|--------|-------------|

## The semantic command layer ŌĆö `CameraActionType` and `applyCommand()`

`CameraInputAdapter` (internal) converts raw input into `CameraActionType` semantic commands and dispatches them via `ICameraManipulator::applyCommand()`. This means:

- **Bindings are decoupled** ŌĆö change which button triggers rotation without editing any manipulator code.
- **Testable without a window** ŌĆö drive a manipulator programmatically by calling `applyCommand()` directly.
- **Replayable** ŌĆö record a sequence of `CameraActionType` + `CameraCommandPayload` events and play them back for deterministic tests.

All 21 action types:

| `CameraActionType` | Meaning | Used by |
|-------------------|---------|---------|
| `eBeginRotate` | Mouse-button-down at cursor position | Orbit, Arcball |
| `eRotateDelta` | Drag delta in pixels | Orbit, Arcball |
| `eEndRotate` | Mouse-button-up | Orbit, Arcball |
| `eBeginPan` | Pan button down | Orbit, Arcball, Ortho |
| `ePanDelta` | Pan drag delta in pixels | Orbit, Arcball, Ortho |
| `eEndPan` | Pan button up | Orbit, Arcball, Ortho |
| `eZoomAtCursor` | Scroll event with cursor position + `zoom_factor` | All |
| `eBeginLook` | Right-mouse-button down for look | FPS, Fly |
| `eLookDelta` | Look drag delta in pixels | FPS, Fly |
| `eEndLook` | Look button up | FPS, Fly |
| `eMoveForward` | W key state | FPS, Fly |
| `eMoveBackward` | S key state | FPS, Fly |
| `eMoveLeft` | A key state | FPS, Fly |
| `eMoveRight` | D key state | FPS, Fly |
| `eMoveUp` | E key state | FPS, Fly |
| `eMoveDown` | Q key state | FPS, Fly |
| `eSprintModifier` | Shift held | FPS, Fly |
| `eSlowModifier` | Ctrl held | FPS, Fly |
| `eOrbitPanModifier` | Shift held in orbit mode (pan-alias) | Orbit, Arcball |
| `eFitBounds` | Programmatic fitToAABB trigger via payload | All |
| `eResetView` | Programmatic reset trigger | All |

`CameraCommandPayload` fields: `x_px`, `y_px`, `delta_x_px`, `delta_y_px`, `zoom_factor`, `pressed`.

**Programmatic orbit without a window:**

```cpp
using namespace vne::interaction;

auto camera = std::make_shared<vne::scene::PerspectiveCamera>(60.0f, 800.0f, 600.0f, 0.1f, 100.0f);
auto manipulator = factory.create(CameraManipulatorType::eOrbit);
manipulator->setCamera(camera);
manipulator->setViewportSize(800.0f, 600.0f);

const double dt = 1.0 / 60.0;
CameraCommandPayload payload;

// Begin drag at viewport center
payload.x_px = 400.0f;  payload.y_px = 300.0f;
manipulator->applyCommand(CameraActionType::eBeginRotate, payload, dt);

// Drag 90┬░ worth of pixels to the right
payload.delta_x_px = 3.0f;  payload.delta_y_px = 0.0f;
for (int i = 0; i < 30; ++i)
    manipulator->applyCommand(CameraActionType::eRotateDelta, payload, dt);

manipulator->applyCommand(CameraActionType::eEndRotate, payload, dt);
camera->updateMatrices();
// camera now has the updated view matrix ŌĆö no GPU or window required
```
