# Git, GitHub & GitLab

Reference and notes for version control. **Git** is the engine; **GitHub** and **GitLab** are the platforms that host and collaborate on it.

## Overview

| Tool | What it does |
|------|--------------|
| **Git** | Local version control. Tracks changes as commits. Runs offline. |
| **GitHub** | Hosting + Pull Requests, Issues, Actions CI/CD. Largest open-source community. |
| **GitLab** | Hosting + Merge Requests, built-in CI/CD, container registry. Can self-host for free. |

---

## Git Concepts

- **Repository** — Folder tracked by Git. Holds files + full history.
- **Commit** — Saved snapshot with a hash and message. Immutable.
- **Branch** — Pointer to a commit. Work in parallel without touching `main`.
- **Staging area** — Prep zone before commit. `git add` puts files here.
- **Remote** — Repo hosted elsewhere (GitHub, GitLab). Push to share; pull to get changes.

## Essential Commands

### Setup

```bash
git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main
```

### Start & clone

```bash
git init                    # Start tracking current folder
git clone <url>             # Clone a remote repo
git clone <url> my-project  # Clone into named folder
```

### Daily workflow

```bash
git status                  # What changed?
git add .                   # Stage all (or git add <file>)
git commit -m "message"     # Commit
git push                    # Send to remote
git pull                    # Get remote changes
```

### Branches

```bash
git branch                  # List branches
git switch -c feature/foo    # Create and switch (modern)
git switch main             # Switch to existing branch
git checkout -b feature/foo # Same as switch -c (older syntax)
git merge feature/foo       # Merge into current branch
git branch -d feature/foo   # Delete branch
```

### Checkout

`git checkout` switches branches or restores files. Prefer `git switch` for branches and `git restore` for files (clearer).

| Command | Does |
|---------|------|
| `git checkout main` | Switch to branch `main` |
| `git checkout -b new-branch` | Create and switch to new branch |
| `git checkout -- file.txt` | Discard changes in file (use `git restore file.txt` instead) |
| `git checkout abc123` | Detached HEAD — view old commit |

### Stash

Temporarily save uncommitted work so you can switch branches or pull cleanly.

| Command | Does |
|---------|------|
| `git stash` | Save changes, working dir clean |
| `git stash pop` | Restore most recent stash (and remove it) |
| `git stash apply` | Restore stash but keep it in list |
| `git stash list` | Show all stashes |
| `git stash drop` | Delete a stash |
| `git stash -m "message"` | Stash with a label |

**Typical use:** You're mid-edit, need to switch to `main` for a quick fix. `git stash` → `git switch main` → fix → `git switch feature/foo` → `git stash pop`.

### Undo

| Command | Does |
|---------|------|
| `git restore <file>` | Discard unstaged changes (irreversible) |
| `git restore --staged <file>` | Unstage file, keep changes |
| `git reset --soft HEAD~1` | Undo last commit, keep changes staged |
| `git revert <hash>` | Create new commit that undoes a commit (safe for shared history) |

:::danger Force-push
`git push --force` rewrites history. Never use on shared branches. Use `--force-with-lease` if you must — it aborts if someone else pushed.
:::

## Rebase

Replay your commits on top of another branch. Creates a linear history.

| Command | Does |
|---------|------|
| `git rebase main` | Rebase current branch onto main |
| `git rebase -i HEAD~4` | Interactive — edit last 4 commits |
| `git rebase --abort` | Cancel rebase |
| `git rebase --continue` | After fixing conflicts |

**Interactive rebase actions:** pick, reword, edit, squash (combine), fixup (combine, drop message), drop (remove).

**Caution:** Don't rebase commits already pushed and shared.

---

## Cherry-pick

Apply one or more specific commits from another branch onto your current branch.

```bash
git cherry-pick abc123           # Apply one commit
git cherry-pick abc123 def456    # Apply multiple
git cherry-pick -n abc123       # Apply but don't commit (stage only)
git cherry-pick --abort         # Cancel
```

**Use case:** Bug fix on `main` needed on `release/v1`. `git switch release/v1` → `git cherry-pick abc123`.

## GitHub

- **Pull Request (PR)** — Propose merging your branch. Review, approve, merge.
- **Issues** — Track bugs and features. Link to PRs.
- **Actions** — CI/CD via YAML in `.github/workflows/`.
- **Pages** — Free static hosting.

:::tip
Use branch protection on `main`: require CI pass + reviewer before merge.
:::

---

## GitLab

- **Merge Request (MR)** — Same idea as PR. Inline test reports, security scans.
- **CI/CD** — Defined in `.gitlab-ci.yml`. Stages, parallel jobs, manual gates.
- **Self-host** — GitLab CE free on your own server.
- **Registry** — Built-in Docker registry per project.

---

## GitHub vs GitLab

| | GitHub | GitLab |
|---|--------|--------|
| Community | Largest | Smaller |
| CI/CD | Actions | Built-in, very flexible |
| Self-host | Paid only | Free CE |
| Best for | Open source, discoverability | Enterprises, DevOps, self-hosting |

---

## Feature Branch Workflow

1. **Branch** — `git switch -c feature/name` from `main`
2. **Commit** — Small commits. Use `feat:`, `fix:`, `docs:` prefixes
3. **Push** — `git push -u origin feature/name`
4. **PR/MR** — Open on GitHub/GitLab, link issues
5. **Review** — CI runs. Teammates review. Fix feedback.
6. **Merge** — Approve, merge, delete branch

:::tip
Keep PRs under ~400 lines. Easier to review.
:::
